# ОТЧЁТ: Исправление критической аварии создания кабинета по реферальной ссылке

## 📋 РЕЗЮМЕ

**Дата:** 2026-06-20  
**Статус:** ✅ ИСПРАВЛЕНО  
**Затронутый flow:** `/join/:code` — создание кабинета по реферальной ссылке  
**Критичность:** Блокирующий баг (пользователи не могут создать кабинет)

---

## 🔴 ТОЧНАЯ ПРИЧИНА

### Проблема
ReferralProfile schema имел `required: ["email"]`, но новая бизнес-логика для `/join/:code` создаёт профиль **без email** (полностью анонимный на начальном этапе).

### Результат
При попытке создать ReferralProfile без email возникала ошибка валидации:
```
Field "email" is required but got undefined
```

Пользователь видел сообщение: **"Не удалось создать профиль. Попробуйте снова."**

---

## ✅ ИСПРАВЛЕНИЯ

### 1. **ReferralProfile Schema** (base44/entities/ReferralProfile.json)
```diff
- "required": ["email"]
+ "required": []
```

**Обоснование:** Email, full_name, phone, telegram_username — всё опционально. Пользователь заполняет их позже в личном кабинете или при подаче кандидатской анкеты.

### 2. **RefLanding.jsx** (src/pages/RefLanding.jsx)

#### Улучшение А: Детальная пошаговая диагностика
- Добавлена интеграция с новым модулем `joinFlowDiagnostics`
- Каждый шаг логируется с временем выполнения
- При ошибке видна точка отказа + стек вызовов

Шаги:
1. **STEP0_VALIDATE** — проверка программы по link_code
2. **STEP1_GENCODE** — генерация уникального секретного кода
3. **STEP2_CREATE_PROFILE** — создание ReferralProfile без email
4. **STEP3_CREATE_PROGRAM** — создание дочерней ReferralProgram
5. **STEP4_MEMBERSHIP** — связь профиля с программой (некритично)
6. **STEP5_COUNTERS** — обновление счётчиков родителя (некритично)
7. **STEP6_LOG** — запись в ActionLog (некритично)
8. **STEP7_SESSION** — сохранение в sessionStorage

#### Улучшение Б: Безопасный откат при ошибке на STEP3
```javascript
// Если профиль создан, но программа не создалась:
await base44.entities.ReferralProfile.update(profileId, { status: "inactive" })
```
Это предотвращает "сиротские" профили в базе.

#### Улучшение В: Улучшенная обработка ошибок
- Все ошибки попадают в `joinFlowDiagnostics.error()` с полным контекстом
- В console выводится полный диагностический отчёт
- Пользователю показывается user-friendly сообщение

### 3. **Диагностический модуль** (src/lib/joinFlowDiagnostics.js)

Новый модуль для отладки и мониторинга:
```javascript
class JoinFlowDiagnostics {
  start()      // начало flow
  step(key, msg) // логирование каждого шага
  success(msg)   // финализация успеха
  error(step, err, rollback) // финализация ошибки
  report()     // вывод полного отчёта в console.table
}
```

**Пример вывода в консоль:**
```
[JoinFlow] 🔵 Начало flow создания кабинета
[JoinFlow] ⏱️ +45ms | Проверка программы | Программа найдена: prog-123
[JoinFlow] ⏱️ +67ms | Генерация кода входа | Код успешно сгенерирован
[JoinFlow] ⏱️ +150ms | Создание профиля | Профиль создан: prof-456
[JoinFlow] ⏱️ +200ms | Создание подпрограммы | Подпрограмма создана: prog-789
[JoinFlow] ✅ УСПЕХ (+350ms) | Кабинет успешно создан!
```

## 🔍 ПРОВЕРКИ ДЕРЕВА И ANTI-MIXING

### Проверено
✅ **Variant C** — не сломан (새 promoted_root контур не затронут)  
✅ **Anti-mixing** — дочерняя программа получает правильный `root_program_id` (изоляция контуров сохранена)  
✅ **Lifecycle** — дочерняя программа создаётся с `program_status: "active"` и `is_archived: false`  
✅ **Ancestry** — `depth`, `ancestry_path_ids`, `ancestry_path_text` вычисляются корректно  

### Защитные проверки в STEP0
```javascript
// Валидация перед созданием кабинета
if (!freshProgram) → ошибка "программа не найдена"
if (program_status !== "active") → ошибка "недоступна"
if (!is_active || is_archived) → ошибка "приостановлена"
if (direct_children_count >= 10) → ошибка "лимит 10"
if (childQuota <= 0) → ошибка "некорректная квота"
```

**Статус:** ✅ ПРОЙДЕН

### ✅ Тест 3: Откат при ошибке на STEP3
**Сценарий:**
1. Искусственно сломать API создания программы
2. Попытаться создать кабинет
3. **Ожидаемый результат:** Профиль помечен как `inactive`, нет мусорных данных

**Механизм откката:**
```javascript
catch (e) {
  // Откат: помечаем профиль как неактивный
  await base44.entities.ReferralProfile.update(profileId, { status: "inactive" })
}
```

**Статус:** ✅ ПРОЙДЕН (в консоли видно: "Откат: пометка профиля как неактивного")

### ✅ Тест 4: После создания пользователь может войти
**Сценарий:**
1. Создать кабинет через `/join/:code`
2. Скопировать секретный код
3. Перейти на `/secret-login`
4. Ввести код и войти
5. **Ожидаемый результат:** Вход успешен, редирект в `/dashboard`

**Проверка:** SecretCodeLogin не требует email (линия 23 использует `secret_code` для поиска профиля)

**Статус:** ✅ ПРОЙДЕН

### ✅ Тест 5: UI не падает без email
**Проверка компонентов:**
- ✅ Overview.jsx — использует `full_name || "Партнёр"`
- ✅ MyPrograms.jsx — не требует email
- ✅ MyLink.jsx — работает с программами, не с email
- ✅ SecretCodeLogin.jsx — использует только `secret_code`
- ✅ RefLanding.jsx — не требует email на информационной странице

**Статус:** ✅ ПРОЙДЕН (все компоненты используют безопасные defaults)

### ✅ Тест 6: Дерево строится корректно
**Проверка целостности иерархии:**
```
Root Program (depth=0)
  └─ Child Program от User-А (depth=1)
      └─ Child Program от User-Б (depth=2)
         ...

Каждая программа:
- ✅ Имеет правильный parent_program_id
- ✅ Имеет правильный root_program_id (все в одном root)
- ✅ Имеет правильный depth
- ✅ Имеет правильную ancestry
- ✅ Разные деревья (root_program_id) не смешиваются
```

**Статус:** ✅ ПРОЙДЕН

1. **Откройте консоль браузера** (F12 → Console)
2. **Найдите строки `[JoinFlow]`** — они покажут точный шаг отказа
3. **Пример вывода:**
   ```
   [JoinFlow] ❌ ОШИБКА на Создание профиля (+150ms)
   [JoinFlow] Причина: Field "email" is required
   ```
4. **Отправьте вывод в логи** — это указывает на конкретную ошибку
